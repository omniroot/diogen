# 00-project-overview.md

## Обзор проекта

### Структура папок

Проект организован в основном вокруг директории `src/`, которая содержит следующий высокоуровневый набор папок:
*   `api/`: Содержит логику для взаимодействия с внешними API (Appwrite, Supabase), общие утилиты для работы с API, определения типов и специфичные файлы запросов (`queries`). Отмечена директория `wow/`, содержащая, по-видимому, экспериментальные или учебные материалы (`gemini-learn-this.md`, `gemini-tutorial.md`, `gemini-work.tsx`, `myturn.tsx`, `test.tsx`, `wow.tsx`), что снижает общую чистоту слоя.
*   `assets/`: Содержит статические ресурсы, в основном иконки.
*   `components/`: Централизованное хранилище для переиспользуемых UI-компонентов. Поддиректория `business/` указывает на разделение между общими UI-компонентами и компонентами, специфичными для бизнес-логики.
*   `features/`: Организация кода по функциональным областям (`habits`, `issues`, `modules`). Каждая фича имеет свои поддиректории для `components` и `modals`, что способствует инкапсуляции.
*   `hooks/`: Содержит пользовательские React-хуки для переиспользуемой логики.
*   `pages/`: Структура, соответствующая маршрутам приложения (`__root.tsx`, `index.tsx`, и поддиректории для основных разделов, таких как `issues/`, `login/`, `posts/`, `projects/`, `test/`, `users/`).
*   `providers/`: Содержит провайдеры контекста React, например, `auth.provider.tsx`.
*   `stores/`: Модули состояния, использующие Zustand, с разделением по областям ответственности (например, `habits.store.tsx`, `header.store.tsx`).
*   `styles/`: Глобальные CSS-файлы.
*   `theme/`: Конфигурация темы, вероятно, для Chakra UI.
*   `utils/`: Общие вспомогательные функции, не привязанные к конкретным слоям.

Корневая директория проекта соответствует стандартной для Vite/React приложения с файлами конфигурации (`package.json`, `tsconfig.json`, `vite.config.ts`, `biome.json` и т.д.) и статическими активами (`public/`).

### Naming (Соглашения по именованию)

Общие соглашения по именованию:
*   **Файлы компонентов:** `PascalCase` (например, `Header.tsx`).
*   **Файлы хуков:** `camelCase` с префиксом `use` (например, `useMedia.tsx`).
*   **Файлы сторов:** `camelCase` с суффиксом `.store.tsx` (например, `habits.store.tsx`).
*   **Файлы API запросов:** Разнообразие (`snake_case`, `kebab-case`, `camelCase`) с суффиксом `.api.ts` или `.api.tsx` (например, `days_records.api.ts`, `posts.api.ts`, но также `appwrite.tsx`). Это указывает на некоторую непоследовательность в именовании файлов, относящихся к API.
*   **Директории:** Преимущественно `kebab-case` или `camelCase`.

В целом, соглашения по именованию в основном последовательны и узнаваемы для проектов на React/TypeScript, за исключением незначительных расхождений в директории `src/api/`.

### Зависимости

Проект использует современный стек технологий для frontend-разработки:
*   **UI-фреймворк:** React.
*   **UI-библиотека:** Chakra UI.
*   **Стилизация:** Emotion (используется Chakra UI), LightningCSS.
*   **Маршрутизация:** TanStack Router.
*   **Управление состоянием:** Zustand (для клиентского состояния), TanStack Query (для серверного состояния и кэширования).
*   **Формы:** React Hook Form, TanStack React Form.
*   **Интерактивность:** @dnd-kit (Drag and Drop), Emoji-mart (выбор эмодзи).
*   **Работа с датами:** `date-fns`, `dayjs`.
*   **Backend-as-a-Service (BaaS) клиенты:** Appwrite, Supabase. Использование двух разных BaaS-клиентов может указывать на постепенный переход или разделение функциональности.
*   **Сборка и типизация:** Vite, TypeScript, `@vitejs/plugin-react`, `babel-plugin-react-compiler`.
*   **Линтинг/Форматирование:** Biome.
*   **Менеджер пакетов/среда выполнения:** Bun.
*   **Прочие утилиты:** `clsx`, `@uidotdev/usehooks`, `@tabler/icons-react`.

### Архитектурный стиль (явный или неявный)

Проект демонстрирует **гибридный архитектурный стиль**, сочетающий элементы **feature-based** организации с **технической/слоевой** структурой.
*   **Feature-based:** Явно прослеживается в директориях `src/features/` и `src/pages/`, где код, относящийся к определенной функциональности или странице, группируется вместе. Это позволяет легче находить и модифицировать код, связанный с конкретной фичей.
*   **Technical/Layer-based:** Существуют выделенные слои для:
    *   **UI:** `src/components/`, `src/assets/`.
    *   **API/Infrastructure:** `src/api/`.
    *   **State Management:** `src/stores/`.
    *   **Hooks/Reusable Logic:** `src/hooks/`.
    *   **Utilities:** `src/utils/`.

Эта комбинация указывает на попытку сбалансировать инкапсуляцию фич с сохранением переиспользуемых технических слоев.

### Предполагаемый масштаб проекта

Исходя из используемых технологий, структуры проекта и разделения ответственности, предполагаемый масштаб проекта — **средний или крупный**.

Признаки:
*   Использование мощных библиотек для маршрутизации (TanStack Router), управления состоянием (Zustand, TanStack Query) и UI-компонентов (Chakra UI) типично для приложений, которые ожидают значительного объема функциональности и сложности.
*   Явное разделение на `features` и `pages` облегчает масштабирование приложения путем добавления новых функциональных модулей без сильного влияния на существующий код.
*   Наличие двух BaaS-клиентов (Appwrite, Supabase) может указывать на сложную интеграцию или стратегию постепенного мигрирования/расширения бэкенда, что характерно для развивающихся проектов.
*   Хорошо организованные слои `api`, `stores`, `hooks` и `components` говорят о попытке создать поддерживаемую и расширяемую кодовую базу, способную к долгосрочному развитию.
*   Наличие `biome.json` и `tsconfig.json` указывает на серьезное отношение к качеству кода и типизации, что важно для больших команд и долгосрочной поддержки.
